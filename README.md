## 3. ที่มาและความสำคัญ (Rationale)
ในปัจจุบันธุรกิจการซื้อขายเสื้อผ้าออนไลน์ (E-Commerce) เติบโตขึ้นอย่างรวดเร็ว แต่ปัญหาสำคัญที่ร้านค้าส่วนใหญ่ต้องเผชิญคือระบบจัดการหลังบ้านที่กระจัดกระจาย การควบคุมสต็อกสินค้าที่ไม่แม่นยำ และขาดการนำข้อมูลมาวิเคราะห์เพื่อเพิ่มยอดขาย แบรนด์ **SALA (ศาลา)** จึงได้คิดค้นแพลตฟอร์มนี้ขึ้นเพื่อรวมระบบการเลือกซื้อสินค้าเสื้อผ้าแฟชั่นที่ไร้รอยต่อ และหน้าจอสรุปข้อมูลสถิติสำหรับผู้ดูแลระบบ (Admin Dashboard) เข้าไว้ด้วยกัน เพื่อเพิ่มประสิทธิภาพในการบริหารคลังสินค้าและการตัดสินใจทางธุรกิจให้เบ็ดเสร็จในแพลตฟอร์มเดียว

## 4. วัตถุประสงค์ของโครงงาน (Objectives)
* เพื่อพัฒนาและออกแบบระบบเว็บแอปพลิเคชัน E-Commerce สำหรับร้านเสื้อผ้าแบรนด์ SALA ให้มีประสิทธิภาพและใช้งานง่าย
* เพื่อออกแบบและพัฒนาระบบหลังบ้านสำหรับ Admin ในการควบคุมสต็อกสินค้าเสื้อผ้าและจัดการหมวดหมู่ได้อย่างถูกต้อง
* เพื่อสร้างระบบรายงานและสรุปข้อมูล (Admin Dashboard) ที่ช่วยให้ผู้ดูแลระบบมองเห็นภาพรวมยอดขายและวิเคราะห์สินค้าขายดีได้อย่างแม่นยำ

## 5. ปัญหาที่พบในปัจจุบัน (Pain Points)

### 5.1 ฝั่งลูกค้า (Customer Pain Points)
* **ปัญหาข้อมูลสต็อกไม่ตรงจริง:** บ่อยครั้งที่ลูกค้ากดสั่งซื้อเสื้อผ้าและชำระเงินไปแล้ว แต่กลับถูกยกเลิกออเดอร์ในภายหลังเนื่องจากสินค้าจริงในคลังหมด หรือไม่มีไซส์ที่ต้องการ เนื่องจากระบบไม่ได้ตัดสต็อกแบบเรียลไทม์ (Real-time)
* **ความยากลำบากในการค้นหาและเลือกซื้อ:** หากร้านค้าไม่มีระบบคัดกรอง (Filter) ที่ดี ลูกค้าจะเสียเวลาอย่างมากในการไล่ดูเสื้อผ้าเพื่อหาไซส์ หมวดหมู่ หรือช่วงราคาที่ตัวเองต้องการ
* **การติดตามสถานะคำสั่งซื้อทำได้ยาก:** ลูกค้าไม่มีหน้าต่างประวัติการสั่งซื้อ (Buy History) หรือช่องทางเช็กสถานะที่ชัดเจน ทำให้ไม่มั่นใจในความปลอดภัยของออเดอร์

### 5.2 ฝั่งผู้ดูแลระบบ (Admin Pain Points)
* **ระบบจัดการหลังบ้านกระจัดกระจาย:** ข้อมูลออเดอร์ รายการสินค้า และข้อมูลลูกค้าไม่ได้ถูกจัดเก็บรวมไว้ในระบบเดียวกัน ทำให้ Admin ต้องทำงานสลับไปมาหลายแพลตฟอร์ม เกิดความล่าช้าและเสี่ยงต่อข้อมูลสูญหาย
* **ความยุ่งยากในการบริหารคลังสินค้า:** การเพิ่ม ลด หรือแก้ไขหมวดหมู่และสต็อกเสื้อผ้าจำนวนมากทำได้ยากและเกิดความผิดพลาดได้ง่ายหากจัดการแบบแมนนวลที่ไม่มีระบบ RDBMS มารองรับ
* **ขาดเครื่องมือวิเคราะห์ยอดขาย:** ขาดหน้าจอสรุปข้อมูลสถิติ (Dashboard) ทำให้ Admin หรือเจ้าของร้านไม่สามารถประเมินภาพรวมรายได้ หรือรับรู้ได้ทันทีว่าเสื้อผ้าชิ้นไหนเป็นสินค้าขายดี ส่งผลให้วางแผนการตลาดและสั่งผลิตสินค้าล็อตถัดไปได้ยาก

## 6. ความต้องการของผู้ใช้งาน (User Requirements)
ระบบมีผู้ใช้งานหลัก 2 กลุ่ม คือ:
* **ลูกค้า (Customer)**
  * สมัครสมาชิก / เข้าสู่ระบบ
  * ค้นหาและเลือกซื้อสินค้าเครื่องแต่งกาย
  * เพิ่มสินค้าลงตะกร้า (Cart)
  * สั่งซื้อสินค้าและชำระเงินจำลอง (Checkout)
  * ดูประวัติการสั่งซื้อ (Buy History)
* **ผู้ดูแลระบบ (Admin)**
  * จัดการข้อมูลสินค้า (เพิ่ม / แก้ไข / ลบ)
  * จัดการหมวดหมู่สินค้า (Category)
  * จัดการผู้ใช้งานในระบบ และตรวจสอบข้อมูลลูกค้า
  * ดูรายงานสถิตียอดขายและข้อมูลวิเคราะห์ผ่าน Dashboard

## 7. ขอบเขตของระบบ (System Scope)
ระบบครอบคลุมฟังก์ชันหลัก 9 ส่วน ตามที่กำหนดไว้ในรายวิชา ได้แก่:
1. การจัดการสมาชิก (Register / Login)
2. การจัดการข้อมูลสินค้า (Product Management)
3. การค้นหาและแสดงรายละเอียดสินค้า (Search & View Products)
4. ระบบตะกร้าสินค้า (Shopping Cart)
5. ระบบสั่งซื้อสินค้า (Order Management)
6. ระบบชำระเงินจำลอง (Simulation)
7. ระบบติดตามสถานะคำสั่งซื้อ (Order Tracking)
8. ระบบจัดการสินค้าและคำสั่งซื้อสำหรับผู้ดูแลระบบ
9. รายงาน / Dashboard สรุปข้อมูลยอดขายเบื้องต้น

## 8. ความสามารถหลักของระบบ (Main Functions) & ความต้องการทางระบบ (System Requirements)

### 8.1 ความต้องการเชิงฟังก์ชัน (Functional Requirements)

**ฝั่งลูกค้า (Customer Portal)**
* **ค้นหาสินค้า (Search & Filter):** ค้นหาและคัดกรองเสื้อผ้าตามชื่อแบรนด์ หมวดหมู่ ราคา และไซส์
* **ตะกร้าสินค้าและการชำระเงิน (Cart & Checkout):** บันทึกรายการสินค้าเสื้อผ้าที่ต้องการซื้อ คำนวณราคารวม กรอกข้อมูลที่อยู่จัดส่ง และเข้าสู่กระบวนการชำระเงินจำลอง
* **การจัดการสมาชิก (Auth):** สมัครสมาชิกและเข้าสู่ระบบเพื่อใช้งานส่วนตัว พร้อมทั้งสามารถเรียกดูประวัติการสั่งซื้อย้อนหลัง (Buy History) ได้

**ฝั่งผู้ดูแลระบบ (Admin Dashboard)**
* **จัดการสินค้าและคลังสินค้า (Product & Stock Management):** อัปโหลดรูปภาพสินค้าเสื้อผ้า คอลเลกชันใหม่ กำหนดราคา ลบสินค้า และอัปเดตจำนวนสต็อกสินค้าในคลัง
* **จัดการหมวดหมู่ (Category Management):** เพิ่ม ลบ หรือแก้ไขหมวดหมู่เสื้อผ้า (เช่น เสื้อยืด, กางเกง, แจ็คเก็ต) ให้สอดคล้องกับสินค้าใหม่
* **รายงานสรุปยอดขาย (Sales Dashboard):** แสดงกราฟและตัวเลขสรุปรายได้รวม สถิติออเดอร์เสื้อผ้าที่ขายดีที่สุด และรายงานยอดคำสั่งซื้อที่รอการจัดส่ง

### 8.2 ความต้องการที่ไม่ใช่เชิงฟังก์ชัน (Non-Functional Requirements)
* **ความปลอดภัย (Security):** ใช้โปรโตคอล HTTPS ตลอดทั้งระบบ, เข้ารหัสผ่านบัญชีผู้ใช้ด้วย Algorithm ที่ปลอดภัย และใช้ระบบ JWT (JSON Web Token) ในการยืนยันสิทธิ์เข้าถึง API
* **ความสามารถในการปรับขยาย (Scalability):** มีโครงสร้างสถาปัตยกรรมระบบและ Database ที่ยืดหยุ่น รองรับการสืบค้นข้อมูลสินค้าได้อย่างรวดเร็วแม้มีปริมาณสินค้าเพิ่มขึ้น
* **ความน่าเชื่อถือและความพร้อมใช้งาน (Reliability & Availability):** ระบบต้องมีความเสถียร สามารถเข้าใช้งานเลือกซื้อเสื้อผ้าได้ตลอด 24 ชั่วโมง และจัดการซอร์สโค้ดของทีมผ่าน GitHub

## 9. เทคโนโลยีที่ใช้ในการพัฒนา (Technology Stack)

| ส่วนประกอบระบบ | เทคโนโลยีที่เลือกใช้ | รายละเอียดและหน้าที่ในการพัฒนา |
| :--- | :--- | :--- |
| **Frontend** | React | ใช้สำหรับสร้างส่วนติดต่อผู้ใช้งาน (User Interface) แบบ Single Page Application (SPA) เพื่อการแสดงผลหน้าเว็บของแบรนด์ที่ลื่นไหล รวดเร็ว และจัดการ State ระบบตะกร้าสินค้าหรือหน้า Dashboard ได้อย่างดีเยี่ยม |
| **Backend** | Node.js | ใช้สำหรับสร้างระบบหลังบ้านและการจัดการ API (Application Programming Interface) เพื่อรองรับคำขอและการประมวลผลข้อมูลต่างๆ ระหว่างหน้าเว็บและฐานข้อมูลได้อย่างมีประสิทธิภาพ |
| **Database** | MySQL | ระบบจัดการฐานข้อมูลเชิงสัมพันธ์ (RDBMS) ใช้สำหรับจัดเก็บข้อมูลที่มีโครงสร้างอย่างเป็นระบบ เช่น ข้อมูลสมาชิก, ข้อมูลรายละเอียดเสื้อผ้า, จำนวนสต็อกสินค้าในคลัง และประวัติการสั่งซื้อ |
| **Styling & Layout** | Tailwind CSS | ใช้สำหรับการจัดแต่งหน้าจอและ UI แบบ Utility-first ช่วยควบคุมดีไซน์เสื้อผ้าแฟชั่นและคอลเลกชันต่างๆ ให้มีความทันสมัย และรองรับการแสดงผลบนอุปกรณ์พกพา (Responsive Design) |
| **Styling & Layout** | Bootstrap | ใช้ควบคู่ในส่วนของ Grid System และ Component สำเร็จรูป (เช่น ฟอร์มกรอกข้อมูล, Modal แจ้งเตือน) เพื่อเพิ่มความเสถียรและโครงสร้างพื้นที่เป็นมาตรฐานของระบบเว็บแอปพลิเคชัน |

## 10. การทดสอบระบบ (System Testing)

### 10.1 ประเภทการทดสอบ (Testing Types)
* **User Acceptance Testing (UAT):** การทดสอบเพื่อความมั่นใจว่าระบบสามารถทำงานได้ตอบโจทย์ความต้องการของผู้ใช้งานจริง ทั้งในบทบาทของลูกค้า (Customer) ที่เข้ามาสั่งซื้อเสื้อผ้าแฟชั่น และบทบาทของผู้ดูแลระบบ (Admin) ที่ต้องบริหารจัดการหลังบ้าน

### 10.2 เครื่องมือที่ใช้ในการทดสอบ (Testing Tools)
* **Manual Testing (Postman):** ใช้เครื่องมือ Postman ในการทดสอบระบบแบบแมนนวล (Manual Test) โดยทำการส่ง Request และตรวจสอบ Response ด้วยตนเอง เพื่อตรวจสอบความถูกต้องของการรับส่งข้อมูลและฟังก์ชันการทำงานต่างๆ ของระบบให้เป็นไปตามที่ออกแบบไว้
