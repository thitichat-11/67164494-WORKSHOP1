## 3. ปัญหาที่พบและต้องการแก้ไข (Pain Points)
* **สำหรับ Customer:** เมื่อซื้อสินค้าแล้วเกิดปัญหา (เช่น ไซส์ไม่ตรง, ได้รับสินค้าล่าช้า) มักติดต่อร้านค้าได้ยาก ข้อมูลการแจ้งเรื่องกระจัดกระจาย และไม่มีการจัดเก็บประวัติการสนทนาที่ตรวจสอบได้
* **สำหรับ Admin:** ขาดระบบ Dashboard สรุปภาพรวมแบบ Real-time ทำให้วิเคราะห์แนวโน้มสินค้าขายดีหรือบริหารจัดการสต็อกเสื้อผ้าในแต่ละคอลเลกชันได้ยาก
* **สำหรับ Staff:** ตกหล่นข้อความหรือคำขอช่วยเหลือของลูกค้าเนื่องจากไม่มีระบบคิว (Ticket) ในการจัดหมวดหมู่และเรียงลำดับความสำคัญของปัญหา

## 4. วัตถุประสงค์ (Objectives)
* เพื่อพัฒนาแพลตฟอร์ม E-Commerce สำหรับจัดจำหน่ายเสื้อผ้าแบรนด์ SALA ที่ใช้งานง่าย ปลอดภัย และเข้าถึงได้ทุกอุปกรณ์
* เพื่อสร้างระบบ Dashboard และการจัดการหลังบ้านที่มีประสิทธิภาพสำหรับผู้ดูแลระบบ (Admin) ในการควบคุมสต็อกสินค้าและดูสถิติ
* เพื่อเพิ่มประสิทธิภาพในการบริการลูกค้าด้วยระบบ Ticket Support ที่ช่วยให้เจ้าหน้าที่ (Staff) ตอบกลับและแก้ไขปัญหาได้อย่างเป็นระบบ

## 5. ขอบเขตของระบบ (Scope)
### ระบบที่รองรับ (In-Scope)
* **ระบบผู้ใช้งาน:** สมัครสมาชิก, ล็อกอิน (Register / Log in) และการจัดการโปรไฟล์
* **ระบบซื้อขายสินค้า:** ค้นหาและกรองสินค้า, ระบบจัดการตะกร้าสินค้า, ระบบชำระเงินจำลอง (Checkout) และระบบบันทึกประวัติการซื้อ (Buy History)
* **ระบบช่วยเหลือ:** ลูกค้าสามารถยื่นข้อร้องเรียน (Submit Ticket) และเจ้าหน้าที่สามารถตอบกลับได้
* **ระบบหลังบ้าน:** การเพิ่ม/ลบ/แก้ไขสินค้าและหมวดหมู่, จัดการข้อมูลลูกค้า และระบบ Dashboard แสดงสถิติ

### ระบบที่ยังไม่รองรับในเฟสนี้ (Out-of-Scope)
* ระบบเชื่อมต่อ Gateway ชำระเงินจริงภายนอก (ในเฟสนี้ใช้ระบบจำลองสถานะหรือแนบหลักฐานสลิป)
* ระบบเชื่อมต่อ API ขนส่งโลจิสติกส์โดยตรงเพื่อคำนวณค่าส่งอัตโนมัติ

## 6. ผู้ใช้งานระบบ (User Roles)
1. **Customer (ลูกค้า):** ค้นหาสินค้า, เพิ่มสินค้าลงตะกร้า, ชำระเงิน, ดูประวัติสั่งซื้อ และยื่น Ticket Support
2. **Staff (เจ้าหน้าที่):** ตรวจสอบรายการร้องเรียนและตอบกลับ Ticket Support ของลูกค้า
3. **Admin (ผู้ดูแลระบบ):** จัดการข้อมูลสินค้า (เพิ่ม/ลบ/แก้ไข), จัดการหมวดหมู่, จัดการบัญชีผู้ใช้งาน และตรวจสอบภาพรวมผ่าน Dashboard

## 7. ข้อกำหนดของระบบ (System Requirements)

### 7.1 User Requirements
* ลูกค้าต้องการค้นหาเสื้อผ้าตามหมวดหมู่ที่ชัดเจน และสั่งซื้อสินค้าเสร็จสิ้นภายในไม่กี่ขั้นตอน
* ลูกค้าต้องการช่องทางติดตามสถานะการขอความช่วยเหลือที่โปร่งใส ตรวจสอบได้ว่ากำลังดำเนินการถึงไหน
* Admin ต้องการแก้ไขข้อมูลสินค้าหรือจำนวนสต็อกได้อย่างรวดเร็วเพื่อไม่ให้กระทบต่อหน้าเว็บ
* Staff ต้องการระบบคิวที่เห็นชัดเจนว่ามี Ticket ใดบ้างที่ยังไม่ได้รับการแก้ไขเพื่อไล่ตอบตามลำดับ

### 7.2 System Requirements
* ระบบต้องทำงานผ่าน Web Browser ในรูปแบบ Responsive Design (รองรับทั้ง Mobile และ Desktop)
* ระบบต้องจัดเก็บข้อมูลในรูปแบบ Relational Database เพื่อรักษาความถูกต้องและความเชื่อมโยงของข้อมูลออเดอร์
* ระบบต้องมีสถาปัตยกรรมแบบแยกส่วน (Decoupled/RESTful API) ระหว่าง Front-end และ Back-end

### 7.3 Functional Requirements
| บทบาท (Role) | ฟังก์ชันการทำงาน (Functional Requirement) |
| :--- | :--- |
| **Customer** | - สมัครสมาชิกและเข้าสู่ระบบ (Register / Log in)<br>- ค้นหาและกรองสินค้าตามชื่อหรือหมวดหมู่เสื้อผ้า<br>- เพิ่ม/ลบ/แก้ไขจำนวนสินค้าในตะกร้าสินค้า<br>- ดำเนินการชำระเงิน (Check out)<br>- เรียกดูประวัติการสั่งซื้อทั้งหมด (Buy History)<br>- ยื่นขอความช่วยเหลือผ่านระบบตั๋ว (Submit Ticket Support) |
| **Staff** | - เรียกดูรายการ Ticket Support ทั้งหมดในระบบ<br>- ตอบกลับข้อความใน Ticket ของลูกค้า<br>- อัปเดตสถานะของ Ticket (Open / In-Progress / Resolved) |
| **Admin** | - เพิ่ม (Add), ลบ (Delete), แก้ไข (Edit) ข้อมูลสินค้าเสื้อผ้า<br>- จัดการและเพิ่มหมวดหมู่สินค้า (Category)<br>- ตรวจสอบและจัดการข้อมูลลูกค้า (เช่น การระงับสิทธิ์ชั่วคราว)<br>- ดู Dashboard สรุปยอดขาย, สินค้าขายดี และสถานะของ Ticket ทั้งหมด |

### 7.4 Non-Functional Requirements
* **Security:** รหัสผ่านของผู้ใช้งานทุกคนต้องได้รับการเข้ารหัส (Hashing เช่น bcrypt) ก่อนบันทึกลงฐานข้อมูล
* **Performance:** หน้าเว็บและระบบค้นหาสินค้าต้องโหลดเสร็จสิ้นภายในเวลาไม่เกิน 3 วินาที เพื่อประสบการณ์ที่ดีของลูกค้า
* **Availability:** ระบบต้องสามารถเปิดใช้งานและรองรับการสั่งซื้อได้ตลอด 24 ชั่วโมง (High Availability)
* **Usability:** ส่วนต่อประสานผู้ใช้ (UI) ต้องมีความทันสมัย เรียบง่าย เข้ากับแบรนด์เสื้อผ้าแฟชั่น และใช้งานง่าย
